- 2026-07-18 — **Phase 1 done.** `types.ts`, `time.ts`, `ephemeris.ts`, `sun.ts`, `trajectory.ts` under `src/lib/astro/` with 31 Vitest tests.
  - Added `time.ts` (not in the original plan file list): the noon→noon window is needed by both `sun.ts` and `trajectory.ts`, so it did not belong in either.
  - **Refraction convention is a real trap.** `SearchAltitude` (twilight) solves the *geometric* angle, while `SearchRiseSet`/`SearchHourAngle` and the charts use *apparent* (refracted) altitude. The two differ by ~0.6° near the horizon — enough to shift horizon events by minutes. `horizontalAt()` now takes an explicit `AltitudeConvention`; `sunAltitude()` defaults to geometric so it agrees with the twilight times. Phase 6 must keep this straight for the custom-horizon crossings.
  - Catalogue objects reach astronomy-engine through `withEngineBody()`, which installs J2000 coordinates into a shared, overwritable `Body.Star1` slot. Callers must not retain the Body past the callback. J2000 → of-date precession is applied (~0.3° error if skipped; there is a regression test pinning this).
  - **Validation caveat:** tests assert physical identities (Polaris altitude ≈ latitude averaged over a day, culmination altitude = 90 − |lat − dec|, twilight ordering, polar day/night at Svalbard and McMurdo) and cross-check our wrapper against the library's independent search paths. They are *not* yet checked against external Stellarium/USNO reference values as the plan's Phase 1 asks. Spot values for manual comparison — Kyiv 2026-10-15: sunset 15:04 UTC, astronomical dusk 16:54, astronomical dawn 02:33, sunrise 04:21; M13 culminates 75.97° at az 179.8°.

- 2026-07-18 — **Phase 2 done.** `src/lib/catalog/` with `catalogs.ts` (registry), `types.ts`, `dso.ts` (load + merge), `solar-system.ts`, `search.ts`, `index.ts` (public API), 36 tests.
  - **Requirement added by the user: the DSO catalog must be extensible** (NGC, Sharpless 2, … later). An object may belong to several catalogs and carry several common names, and must be findable by any of them. The model reflects that: `CatalogObject` has `designations: Designation[]` and `names: string[]` rather than a single id/name. Adding a catalog = register it in `CATALOGS` + drop a JSON file in `data/` + import it in `dso.ts`; nothing else changes.
  - Objects appearing in two data files are **merged by shared designation** (the only cross-catalog identity available), so adding NGC later will not duplicate the ~40 Messier objects it contains. First file listed wins on coordinates; designations and names are unioned.
  - Data is **generated, not hand-written**: `npm run catalog:build` (`scripts/build-catalog.mjs`) pulls OpenNGC's `NGC.csv` + `addendum.csv` and emits `data/messier.json`. Positions spot-checked against published J2000 values in tests. **OpenNGC is CC-BY-SA-4.0 — the app must show the attribution** (`catalogSources` exposes it; wire into the credits in Phase 7). See `data/README.md`.
  - **M102 is resolved as NGC 5866** (galaxy in Draco, mag 9.9), not as OpenNGC has it — OpenNGC treats M102 as a re-observation of M101. Full 110 objects. The departure lives in `MESSIER_OVERRIDES` in the generator; the generator now *throws* if two rows claim one designation instead of merging them silently, so any future conflict has to be resolved explicitly.
  - Search ranks exact designation > common name > bare catalog number > name substring > object type, tie-broken towards the brighter object. Accepts `M13`, `m 13`, `messier 13`, `NGC6205`, `ngc 6205`, bare `6205`, and name fragments.
  - Solar system objects live in `solar-system.ts` (Mercury…Neptune + Moon, ready for Phase 8) and are searchable through the same index. The Sun stays in `astro/sun.ts` — it drives twilight, it is not a target.
- 2026-07-18 — **Phase 3 done.** `src/lib/horizon/` (`horizon.ts`, `parser.ts`, `index.ts`) and `src/lib/observatory/` (`types.ts`, `store.ts`, `index.ts`), plus `LocationInput`, `HorizonUpload` and `ObservatoryManager` under `src/components/`. 50 new tests, 117 total.
  - The user's own horizon file (`.plan/e.c.carr-horizon.txt`, 12 points) is checked in as a test fixture at `src/lib/horizon/fixtures/` and imported with `?raw` — the parser is tested against a real file, not only synthetic input.
  - **Azimuth is circular and the wrap segment is the easy thing to get wrong.** The Carr horizon's points run 5°…335°, so due north falls in the 335°→5° gap and interpolates to 53.3°, not 0°. `Horizon.altitudeAt` straightens that segment by lifting its end past 360°; there are continuity and out-of-range-azimuth tests pinning it.
  - The parser reports issues with **line numbers** rather than failing or silently dropping bad rows, so `HorizonUpload` can point at the line to fix. A trailing `360 x` row folds onto 0°; a contradictory duplicate azimuth keeps the first value and warns.
  - **Observatories store the raw horizon text, not parsed points** — an imported file survives localStorage byte for byte, comments included, and the user re-edits exactly what they pasted. Parsing happens on the render path via `horizonFromText`, which memoizes the last text so redraws don't re-parse.
  - Store invariants: the list is never empty and one observatory is always selected; deleting the last one recreates the default (Greenwich). Corrupt JSON, entries failing validation, a dangling `selectedId`, absent localStorage and a throwing `setItem` are all handled and tested. `ObservatoryStore` is plain TS implementing the Svelte store contract, so it is testable under Node while components read it as `$observatories`.
  - **Verification gap closed the same day** — see the UI testing entry below.
  - `App.svelte` is still a placeholder shell: it now mounts `ObservatoryManager` and prints the interpolated horizon altitude at the eight cardinal points, so the phase is manually exercisable until the charts land.
- 2026-07-18 — **UI testing added** at the user's request: `@testing-library/svelte` 5 + `@testing-library/user-event` + jsdom. 44 component tests, 161 total.
  - **Vitest now runs two projects** (`vitest.config.ts`): `unit` (Node, `src/lib/**/*.test.ts`) and `components` (jsdom + `src/test/setup-dom.ts`, `src/components/**/*.test.ts`). Splitting keeps the astronomy tests in Node — faster, and they cannot accidentally lean on a browser global. Project membership follows the **directory**, so no `*.svelte.test.ts` naming that could be confused with Svelte's `.svelte.ts` rune modules. `npm test` runs both; `test:unit` / `test:components` run one.
  - `cleanup` is registered explicitly in the setup file: Vitest runs without `globals`, so testing-library cannot self-register its `afterEach` and components would otherwise stay mounted between tests.
  - **`ObservatoryManager` now takes an optional `store` prop** (defaulting to the singleton) instead of reaching for the singleton directly, so each test gets an isolated store over `MemoryStorage` — extracted to `lib/observatory/memory-storage.ts` and shared with the store tests. The component reads the store through a rune bridge (`$state` + `$effect(() => store.subscribe(...))`, the subscription's unsubscribe being the effect cleanup) rather than `$store` auto-subscription, which does not work on a prop.
  - **The component tests immediately found a real bug.** Under Vitest's jsdom, `window === globalThis` and Node 26's *experimental* `localStorage` shadows jsdom's: reading it prints a warning and yields `undefined`. `defaultStorage()` trusted the type and handed back `undefined`, so the app-wide store silently persisted nothing. It now checks that what it got actually has `getItem`, and the DOM setup installs a real in-memory `Storage` on `globalThis` so component tests exercise a browser-like environment. A test covers the default (no-prop) wiring writing to `localStorage`, which is the path the real app takes.
  - Tests were mutation-checked rather than assumed: deleting the post-save re-read fails 1 test, disabling the horizon wrap-around segment fails 6.
